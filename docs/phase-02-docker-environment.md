# Phase 2: Docker Environment Setup

## 🎯 Objective of This Phase

**In simple language:**
We're going to set up ALL the services our pipeline needs — databases, Kafka, MinIO, monitoring tools — using Docker. Think of Docker as a "box" that contains everything a service needs to run. Instead of installing PostgreSQL, Kafka, etc. on your machine manually, we define them in a file and start them all with ONE command.

**In technical language:**
This phase configures a multi-container Docker Compose environment that provisions all infrastructure services required by the CDC pipeline. Each service is isolated in its own container with defined resource limits, health checks, networking, and volume persistence. This mimics a production deployment topology on a single development machine.

## 📚 Concepts You Need to Understand First

### 1. What is Docker?

**Simple explanation:**
Docker is like a shipping container for software. Just like a real shipping container can hold anything and works on any ship/truck/train, a Docker container holds your application and works on any computer. It packages the code + dependencies + configuration into one portable unit.

**Technical explanation:**
Docker is a containerization platform that uses OS-level virtualization to deliver software in packages called containers. Containers are isolated from each other and from the host system, sharing only the kernel. They start in seconds (unlike VMs) and use minimal resources.

### 2. What is Docker Compose?

**Simple explanation:**
If Docker runs ONE container, Docker Compose runs MANY containers together. It's a recipe file (YAML) that says "I need a PostgreSQL, a Kafka, a MinIO, etc. — start them all and connect them together."

**Technical explanation:**
Docker Compose is a tool for defining and running multi-container Docker applications. You define services, networks, and volumes in a `docker-compose.yml` file, then start the entire stack with `docker compose up`. It handles container orchestration, dependency ordering, networking, and volume management.

### 3. What is a Docker Volume?

**Simple explanation:**
Containers are temporary — when you delete them, all data inside is lost. A volume is like an external hard drive that persists even when the container is destroyed. We use volumes for database data so we don't lose everything on restart.

**Technical explanation:**
Docker volumes are the preferred mechanism for persisting data generated by containers. They are completely managed by Docker and are independent of the container lifecycle. Named volumes can be shared among containers and survive `docker compose down`.

### 4. What is a Docker Network?

**Simple explanation:**
By default, containers can't talk to each other. A Docker network creates a private "LAN" where containers can find each other by name. When our Python consumer wants to talk to Kafka, it just uses the name "kafka" — Docker resolves it to the right IP.

**Technical explanation:**
Docker Compose creates a default bridge network for all services in the compose file. Containers on the same network can resolve each other by service name (DNS). We define custom networks to isolate traffic (e.g., separate database traffic from monitoring traffic).

### 5. What is a Health Check?

**Simple explanation:**
A health check is like a doctor's checkup for containers. Docker periodically asks each container "are you alive and working?" If a container says "no" (or doesn't respond), Docker marks it unhealthy. Other containers that depend on it won't start until it's healthy.

**Technical explanation:**
Health checks are commands that Docker runs inside a container at configured intervals to determine if the service is ready to accept requests. The `depends_on` condition `service_healthy` ensures proper startup ordering — a consumer won't start until Kafka reports healthy.

### 6. What are Port Mappings?

**Simple explanation:**
Containers are isolated — you can't access them from your browser/terminal by default. Port mapping creates a "door" from your machine into the container. `5432:5432` means "my machine's port 5432 connects to the container's port 5432."

**Technical explanation:**
Port mappings (`host:container`) expose container ports to the host machine. This enables accessing services from the host (e.g., connecting to PostgreSQL via `localhost:5432`). Only exposed ports are accessible from outside the Docker network.

## 📄 docker-compose.yml — Detailed Explanation

The compose file is organized in layers matching our architecture:

```
1. SOURCE LAYER      → postgres-source
2. STREAMING LAYER   → zookeeper, kafka
3. STORAGE LAYER     → postgres-target, minio
4. MONITORING LAYER  → prometheus, grafana
5. NETWORKS          → cdc-network (shared communication)
6. VOLUMES           → Named volumes for data persistence
```

### Why this ordering matters:
- Kafka depends on Zookeeper → Zookeeper starts first
- Our consumer (Phase 7) depends on Kafka → Kafka must be healthy first
- Target PostgreSQL is independent → Can start in parallel
- Monitoring observes everything → Can start anytime

## ⚠️ Common Mistakes and Debugging Tips

| Problem | Cause | Fix |
|---------|-------|-----|
| "port already in use" | Another service using that port | Change host port in .env or stop conflicting service |
| Container keeps restarting | Misconfiguration or dependency not ready | Check logs: `docker logs <container-name>` |
| Kafka can't connect to Zookeeper | Zookeeper not healthy yet | Add `depends_on` with health check condition |
| "permission denied" on volume | Windows/Linux permission mismatch | Use named volumes instead of bind mounts |
| MinIO won't start | Port 9000 used by another app (common on Mac) | Change MINIO port to 9002 in .env |
| Containers can't find each other | Not on the same network | Ensure all services are on `cdc-network` |
| PostgreSQL says "role does not exist" | Init script didn't run | Remove volume and recreate: `docker compose down -v && docker compose up -d` |
| Kafka says "broker not available" | KAFKA_ADVERTISED_LISTENERS misconfigured | Ensure both INTERNAL and EXTERNAL listeners are set |

### Debugging Commands:

```bash
# See all container logs
docker compose -f docker/docker-compose.yml logs

# See specific container logs
docker logs postgres-source --tail 50

# Enter a container's shell
docker exec -it kafka bash

# Check container resource usage
docker stats

# Remove everything and start fresh
docker compose -f docker/docker-compose.yml down -v
docker compose -f docker/docker-compose.yml up -d
```

## 🎤 Interview Questions for This Phase

### Beginner Level:

1. **Q:** What is the difference between Docker and a Virtual Machine?
   **A:** Docker containers share the host OS kernel and isolate only the application layer (lightweight, starts in seconds). VMs virtualize the entire OS including kernel (heavy, starts in minutes). Containers use ~MB of memory; VMs use ~GB.

2. **Q:** What is Docker Compose used for?
   **A:** Docker Compose defines and manages multi-container applications. It uses a YAML file to configure services, networks, and volumes, then starts/stops the entire stack with a single command.

3. **Q:** Why do we use named volumes instead of bind mounts for databases?
   **A:** Named volumes are managed by Docker, are portable, have better performance on non-Linux hosts, and don't create permission issues. Bind mounts couple the container to a specific host path.

### Intermediate Level:

4. **Q:** How do you ensure Kafka starts only after Zookeeper is ready?
   **A:** Using `depends_on` with a `condition: service_healthy` constraint. Zookeeper's health check verifies it's accepting connections. Only when the health check passes does Docker start Kafka.

5. **Q:** What is the difference between `KAFKA_LISTENERS` and `KAFKA_ADVERTISED_LISTENERS`?
   **A:** `LISTENERS` defines what interfaces/ports Kafka binds to inside the container. `ADVERTISED_LISTENERS` defines what address clients are told to connect to. For Docker, we need both an internal listener (for container-to-container) and an external listener (for host access).

6. **Q:** How would you handle secrets (passwords) in a production Docker deployment?
   **A:** In production, use Docker Secrets (Swarm), Kubernetes Secrets, or external secret managers (HashiCorp Vault, AWS Secrets Manager). Never hardcode secrets in docker-compose.yml or check .env files into version control.

### Advanced Level:

7. **Q:** How would you scale this Docker setup for production?
   **A:** Replace Docker Compose with Kubernetes. Use StatefulSets for Kafka/Zookeeper, Deployments for stateless services, PersistentVolumeClaims for storage, and Helm charts for templating. Add an ingress controller, service mesh, and proper resource limits.

8. **Q:** What happens to Kafka data when you run `docker compose down` vs `docker compose down -v`?
   **A:** `down` removes containers and networks but preserves volumes (data survives). `down -v` also removes volumes (all data is destroyed). In development, `-v` gives a clean slate. In production, volumes must be backed up before destruction.
